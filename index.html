Strategic frame

This is a pilot decision, not a platform decision

The common mistake is to reward feature richness too early. For this phase, a vendor is valuable only if it helps Expatrio answer the behavioral question quickly: does story exposure lead to CTA clicks, insurance switching, or reduced private-insurance cancellation?

1

Measure behavior

Story exposure must be joinable to downstream insurance outcomes. Vendor dashboards alone are not enough.

2

Control exposure

Expatrio needs a clean cohort and holdout. The safest pattern is to gate the widget internally.

3

Move fast

The pilot should launch with minimal engineering and no long procurement cycle.

4

Keep exit clean

Own the event schema and content assumptions so the vendor can be replaced after the test.

Cost and effort

Cost matters because the pilot is buying evidence

Paying more can be rational after the format works. Before proof, cost should be judged by learning efficiency: how cheaply and quickly can Expatrio test the behavior change?

Best pilot economics

StorifyMe

Low public entry price and web-first embed make it the practical first move, provided scale pricing is clarified early.

Best capability, higher cost

Storyly

Stronger product depth, but the cost and sales process are heavier than the first pilot requires.

Best leverage

StorySDK / OSS spike

The spike creates a real fallback and gives procurement leverage if a vendor tries to reprice after pilot success.

Build vs buy

Own the learning layer, rent the rendering layer

The safest architecture is simple: vendor SDKs can render the stories, but Expatrio must own eligibility, holdouts, event schema, analytics joins, and the post-pilot decision record.

1

Create one wrapper component

<StoryRail /> owns placement and loading. Vendor-specific code stays inside the wrapper.

2

Gate exposure internally

Feature flags and user attributes decide eligibility. The vendor should not own the experiment design.

3

Emit Expatrio-owned events

Use a fixed schema: story_impression, story_open, story_complete, story_dismiss, story_cta_click.

4

Keep vendor selection replaceable

If the event contract is owned by Expatrio, switching StorifyMe → Storyly → StorySDK → internal becomes manageable.

Contrarian point

The expensive part of building is not the story player

Rendering is relatively cheap. The real cost is the CMS, media pipeline, scheduling, analytics dashboards, privacy review, QA, and making marketing independent. That is why a short SaaS pilot plus OSS spike is better than jumping straight to internal build.

Decision rule after pilot Keep buying only if the vendor’s scale pricing is lower than the known build-and-run cost and the vendor contributes more than rendering.

Next questions

Questions to ask before committing

These questions should be answered in writing before Expatrio gives any vendor a long-term advantage.

  1. Controlled rollout: Can Expatrio decide exactly which users see the widget, without sending PII?
  2. Event callbacks: Can we capture impression, open, completion, skip/dismiss, replay, and CTA click?
  3. Attribution: Can events include story_id, cohort_id, pseudonymous user_id, timestamp, and CTA target?
  4. Analytics export: Can events be sent to Amplitude or exported raw?
  5. Embed model: Is the web embed iframe-based or native DOM?
  6. Pricing: What is the written cost at 5k, 25k, 50k, and 100k targeted users?
  7. Metering: What exactly is billable: exposed user, MAU, impression, story view, or video play?
  8. Privacy: Can the vendor provide DPA, SCCs, subprocessor list, residency details, and deletion terms?
  9. Exit: Can Expatrio export stories and analytics history if it leaves?
  10. Commitment: Can the pilot run without annual commitment?
